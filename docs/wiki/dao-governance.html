DAO Governance Phase 5 — Planned

Two-chamber governance system via Telegram — on-chain anchoring, off-chain voting.

Executive Summary

IFR governance happens where the community lives: Telegram. A hybrid on-chain/off-chain system respects IFR's burning characteristic (no transaction costs for voting) while providing maximum transparency and security.

The system consists of two complementary chambers:

  1. Der Rat (The Council) — Team governance for strategic decisions and capital allocation.
  2. Das Forum (The Forum) — Community innovation with built-in economic spam protection.

Architecture

Three-layer design: Presentation (Telegram), Logic (Bot Backend), Truth (Blockchain).

Chamber 1: Der Rat (The Council)

Team Governance

Purpose: Protocol parameters, treasury allocation, strategic partnerships.

Participants: Dev team / Multisig signers.

Proposal rights: Team only.

Process

  1. Team member creates proposal via /rat_proposal "Title" "Description" "Duration"
  2. Bot posts proposal in @IFR_Rat (read-only for community). Discussion in @IFR_Forum.
  3. Snapshot voting: Only users with IFR-Lock participate. Off-chain votes weighted by lock amount (1 IFR = 1 vote).
  4. Finalization: Bot aggregates weighted results, creates Merkle Tree, sends single on-chain TX to RatVoting.sol: ProposalID + ForWeight + AgainstWeight + MerkleRoot.

Every voter can cryptographically prove their vote was included in the Merkle Tree.

Chamber 2: Das Forum (Community Innovation)

Community Governance

Purpose: Community brings ideas; most popular ones go to binding vote.

Participants: Anyone with IFR-Lock.

Proposal cost: 10 IFR (spam protection — refunded on success, burned on failure).

Process

  1. Submit: /propose My Idea in @IFR_Forum → Bot checks lock, transfers 10 IFR to IFRSpamProtection.sol
  2. Voting period (7 days): Community votes via reactions. Bot weights each reaction by voter's IFR-Lock amount. Auto-deletion if downvotes > 50%.
  3. Ranking: Daily cron calculates scores. Top 3 proposals posted daily.
  4. Consensus phase: Proposal stays 10 days in Top 3 → EIP-712 binding vote. Users sign via /sign 42 yes.
  5. Finalization: Bot bundles signatures → single TX to ForumVoting.sol. Contract verifies signatures + lock status. Result stored on-chain permanently.

Bot Architecture

A single bot serves both chambers with strict role separation.

ComponentFunction
Wallet-MapperEncrypted DB: TelegramUserID ↔ EthereumAddress (one-time free signature)
IFR-ReaderReads IFR Lock Contract permanently (Infura/Alchemy), caches lock amounts
Reaction-TrackerTracks every reaction with timestamp + lock weight of user
Poll-ManagerCreates native Telegram polls for Council, aggregates weighted results
Cron: RankingRuns daily, calculates top proposals
Cron: ConsensusChecks if proposal stayed 10 days in Top 3
Cron: FinalizerSends collected votes on-chain after signature phase
ExecutorSole component with private key / multisig access for on-chain TXs

Smart Contracts

RatVoting.sol

Stores results of team votes (snapshot model).

  • finalizeProposal() — Bot-only, stores ProposalID + ForWeight + AgainstWeight + MerkleRoot
  • verifyVote(proposalId, voter, vote, weight, proof) — View function for vote verification

ForumVoting.sol

Manages binding community votes (signed-vote model).

  • submitVotes(proposalId, votes[], signatures[]) — Verifies signatures (ecrecover), checks IFR Lock, stores final result

IFRSpamProtection.sol

Manages the 10 IFR proposal fee.

  • Successful consensus → 10 IFR refunded (incentive for good proposals)
  • Downvoted > 50% → 10 IFR burned (deflationary)
  • Requires approve() from user before submission
